Context Signals MCP — Benchmark Findings

Fixture Benchmarks + Real-World Testing — April 2026

What Is Context Signals MCP?

Context Signals MCP is an experimental local-first MCP server for pre-indexed code navigation.

It extracts compact structural metadata such as functions, routes, classes, imports, files, and line numbers, then exposes those signals to coding agents through MCP.

It is most useful for long-lived, evolving codebases where agents repeatedly ask navigation and discovery questions.

Scope: It is not a replacement for reading source code, LSP, or implementation-level reasoning.

Summary

Context Signals MCP shows strong benefits for warm-cache navigation queries, but can perform worse on cold start, small projects, or implementation-level questions.

Context Signals MCP was tested across benchmark fixtures and two real local projects. The results show conditional value: when signals are pre-indexed and the query is navigation/discovery focused, the MCP can reduce query-time context significantly. However, cold-start usage, small projects, and implementation-level questions can add overhead and perform worse than baseline.

Real-World Testing Findings

In addition to controlled framework fixtures, Context Signals MCP was tested on two local projects: a small Todo API and the larger PhotoVerify API. These tests showed that the MCP provides value only when the signal store is warm and the query is navigation/discovery focused.

Scenario Result Interpretation
PhotoVerify cached navigation query 80.2% context reduction Strong benefit when useful signals are already indexed
PhotoVerify cached route query 39.9% context reduction Moderate benefit when signals reduce but do not eliminate reads
PhotoVerify cold search ~10x worse Cold search added overhead and then fell back to file reads
Todo API, 5 simple queries 65.6% worse overall Small project and few queries did not amortize indexing/search overhead

When It Works Best

  • Signals are pre-indexed before users ask questions.
  • The project is medium or large.
  • The user asks repeated navigation/discovery questions.
  • The codebase is long-lived and evolves over time.
  • The agent needs file, symbol, route, handler, or line-number metadata.
  • The workflow benefits from a persistent local code-navigation memory.

When It Does Not Work Well

  • Cold start with an empty signal store.
  • Small projects where direct file reads are already cheap.
  • Single or very few queries where indexing cost cannot be amortized.
  • Implementation-level reasoning questions.
  • Queries that require full business logic, validation rules, or edge-case analysis.
  • Signal store contains only file paths instead of real function, route, class, and import signals.
  • The workflow reads files and then ingests separately, causing duplicate overhead.

Cold Start vs Warm Cache

Mode What Happens Expected Result
Cold Start No signals exist yet. Search may fail and fall back to reads. Can be worse than baseline.
Warm Cache Signals are already indexed and reusable. Can reduce context for navigation queries.
Incremental Update Only changed files are re-indexed. Target design for long-lived projects.

Correct Workflow vs Incorrect Workflow

Works Well

1. Index project once
2. Store function / route / class / import signals
3. Reuse signals for repeated navigation queries
4. Read source code only when implementation details are needed

Does Not Work Well

1. Search signals before indexing
2. Get empty signal result
3. Fall back to broad file reads
4. Pay both MCP overhead and source-read cost

Problem Statement

Coding agents often spend context discovering where things are before they can reason about implementation details. In long-lived codebases, the same navigation questions appear repeatedly: where is this route, where is this handler, which file owns this logic, what endpoints exist?

Reading full files every time is wasteful when the question only needs structural metadata. Context Signals MCP addresses this by creating a reusable local signal store for code-navigation metadata.

How We Solve It

Context Signals MCP shifts the workflow from repeated source-file discovery to reusable structural lookup.

First, it indexes code structure into local signals. Later, agents can query those signals to find files, routes, functions, classes, and line numbers. For implementation-level tasks, the signal result should be used as a locator before reading the relevant source code range.

Important: Signals are a map, not the territory. Source code remains the ground truth.

High Level Architecture

Initial run:
  Source files → Scanner → Extractor → Local signal store

Later navigation query:
  User query → Signal search → File/route/function metadata → Optional targeted source read

For warm-cache navigation queries, agents can query signals before reading files → lower query-time context

Signals are extracted once per session and stored locally. Subsequent queries hit the signal store rather than source files.

Break-Even Reality

Context Signals MCP has an upfront indexing cost. The system becomes useful only when that cost is amortized across repeated queries.

In controlled fixture benchmarks, break-even was estimated around 62 navigation queries. In real testing, small projects with only a few queries performed worse overall.

This means Context Signals is better suited for long-lived projects and repeated agent workflows than one-off questions.

Signal Quality Matters

Real testing showed that file-path-only signals are not enough. Useful navigation requires extracting real code structure:

  • Function names
  • Route definitions
  • Class names
  • Imports
  • Handlers/controllers
  • File path and line number

Without these, signal search may return empty or low-value results and fall back to expensive file reads.

Limitations

  • Cold start can be worse than baseline if signals are not pre-indexed.
  • Small projects may not benefit because direct reads are already cheap.
  • Single-query usage may not amortize indexing cost.
  • File-path-only signals are insufficient for route/function discovery.
  • Current workflow should avoid separate read + ingest duplication where possible.
  • Real-world savings depend on how the agent chooses between signal search and source reads.
  • Implementation-level reasoning still requires reading source code.
  • Structure extraction may not capture dynamic runtime behavior.
  • LSP enrichment depends on local language server availability.
  • Results vary by repository structure, framework, and query type.

v1.0 Lessons Learned

  • Warm-cache navigation can reduce context significantly.
  • Cold start can be worse than baseline.
  • Small projects may not benefit.
  • Signal quality is more important than signal quantity.
  • Metadata helps navigation, but source code remains required for reasoning.
  • The best use case is persistent project memory for long-lived agent workflows.

Future Enhancements

1. Auto-Indexing

Automatically scan and index project structure before the first search, or trigger indexing when the signal store is empty.

2. Incremental Indexing

Re-index only changed files instead of rescanning the whole project.

3. Better Signal Extraction

Extract real functions, routes, classes, imports, handlers, and framework-specific structures instead of file paths only.

4. AST-Based Extraction

Use AST parsing for JavaScript/TypeScript to reduce false positives and improve line-level accuracy.

5. Optional LSP Enrichment

Use LSP where available for references, definitions, diagnostics, and symbol metadata.

6. Query Intent Detection

Detect whether the user is asking a navigation question or an implementation reasoning question.

7. Targeted Source Reads

For implementation queries, use signals to locate the relevant file and line range, then read only that region.

8. Warm-Cache Benchmarking

Separate cold-start, warm-cache, and incremental-update benchmarks.

9. Improved Baseline Modeling

Compare against real agent traces instead of simplified baseline simulations.

10. Larger Project Testing

Test on larger open-source repositories and real-world monorepos.

Conclusion

Context Signals MCP shows conditional but meaningful value.

It is most useful as a pre-indexed navigation cache for long-lived, evolving codebases where agents repeatedly ask discovery questions.

Real testing shows strong gains for warm-cache navigation queries, but also shows clear failure modes: cold start, small projects, file-path-only signals, and implementation-level reasoning.

The next version should focus on auto-indexing, better structural extraction, incremental updates, and targeted source reads.
