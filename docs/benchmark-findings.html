Problem Statement

Context windows are finite resources. When working with large codebases, AI agents must navigate to find functions, routes, handlers, and other structural elements before they can reason about implementation details.

Current approaches require reading full source files to extract this structural information. This consumes significant context tokens without proportional value—the agent only needs metadata like function names, file locations, and line numbers, not the full implementation.

This inefficiency accelerates context window exhaustion, limits the effective codebase size an agent can work with, and reduces overall agent productivity. The core challenge: how do we provide agents with structural context without overwhelming their context windows?

How We Solve It

We extract compact signals from code structure—function declarations, route definitions, class definitions, and imports—storing only the metadata agents need for navigation.

When an agent queries instead of reading files, we retrieve relevant signals from our index, providing precise locations without consuming context on implementation details.

This improves retrieval precision by providing structured entry points into the codebase, reducing reliance on heuristic file search and broad context loading.

This shifts cost: from repeated full-file reads to one-time indexing plus efficient navigation queries. The upfront indexing cost is amortized across repeated queries.

Scope: Signal queries return structural metadata only. Implementation-level reasoning still requires reading source code. Benefits apply to navigation and discovery queries.

Signal Extraction Approach

Signal extraction uses framework-specific parsers to identify code structure: functions, classes, routes, imports, and handlers. The extraction is designed to be accurate for each framework's idiomatic patterns (Express middleware chains, Fastify plugins, Next.js file-based routing).

Break-Even Analysis

Current Result: ~62 queries to recoup indexing cost

Break-even is sensitive to baseline assumptions. Real-world agents may reuse context, reducing the effective break-even threshold. The baseline simulation may be optimistic.

We encourage community testing with varied codebase sizes and query patterns to validate whether this break-even point accurately reflects production scenarios.

Tool Calls Reduction

Current Result: Baseline simulation shows 0% reduction

Our baseline simulation models agent file reads as single operations. Real agents typically make multiple reads (grep, glob, read) per query. This makes our baseline appear more efficient than real agents, inflating break-even estimates.

We target improved baseline modeling in v1.1 to better reflect actual agent tool usage patterns.

Transparency: We err on the side of under-promising. The core metrics (query context reduction, accuracy, hit rate) are measured objectively. Break-even and tool calls are published with methodology for validation.
